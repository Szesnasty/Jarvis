# Step 24c — MCP client integration & evaluation harness

> **Parent:** [step-24-mcp-server.spec.md](step-24-mcp-server.spec.md)
> **Depends on:** 24a (backend), 24b (frontend)
> **Goal:** Prove the cost & quality claims with a reproducible benchmark,
> and ship copy-paste configs for the four most likely client apps.

The whole MCP project only matters if external agents (Claude Desktop,
Cursor, Continue, Zed) actually save tokens **and** answer at least as
well as the raw-paste baseline. This step delivers:

1. Reference client configs (verified by manual smoke tests).
2. An automated **eval harness** that scores a fixed task suite under
   two conditions — *baseline* (raw context dump) vs *MCP* (tool-using
   agent) — and emits a Markdown report.
3. Documentation in `docs/features/mcp-server.md`.

---

## Reference client configs

All configs live in `docs/features/mcp-server/clients/` so they're
versioned and can be regenerated by the snippet generator (24b §3).

### `claude-desktop.json`

```json
{
  "mcpServers": {
    "jarvis": {
      "command": "python",
      "args": ["-m", "services.mcp", "--transport", "stdio"],
      "cwd": "/ABSOLUTE/PATH/TO/Jarvis/backend",
      "env": {
        "JARVIS_WORKSPACE": "/ABSOLUTE/PATH/TO/jarvis-workspace"
      }
    }
  }
}
```

Drop in `~/Library/Application Support/Claude/claude_desktop_config.json`
on macOS. Restart Claude Desktop. `jarvis` should appear in the tools
panel with all 15 read tools (or 18 if `mcp.allow_writes=true`).

### `cursor.json`

`~/.cursor/mcp.json`:

```json
{
  "mcpServers": {
    "jarvis": {
      "url": "http://127.0.0.1:8765/sse",
      "headers": { "Authorization": "Bearer <PASTE_TOKEN_FROM_JARVIS_SETTINGS>" }
    }
  }

```

Requires the SSE server to be running (`POST /api/mcp/start` from
the Jarvis UI).

### `zed.json`

```jsonc
{
  "context_servers": {
    "jarvis": {
      "command": {
        "path": "python",
        "args": ["-m", "services.mcp", "--transport", "stdio"],
        "env": { "JARVIS_WORKSPACE": "/ABSOLUTE/PATH/TO/jarvis-workspace" }
      }
    }
  }

```

Each config has a matching **manual smoke test** in
`docs/features/mcp-server/clients/SMOKE-TESTS.md`:

1. Install config, restart client.
2. Ask: *"Use jarvis to find my recent notes about the onboarding bug."*
3. Verify: client invokes `jarvis_search_memory` with `query="onboarding bug"`,
   results render, and the assistant cites paths.
4. Repeat with one of: `jarvis_jira_describe_issue`, `jarvis_graph_query`,
   `jarvis_session_recent_decisions`.

Smoke tests are **manual** (these clients can't be scripted reliably);
the eval harness below provides the automated coverage.

## Eval harness

Lives in `backend/scripts/mcp_eval/`. Standalone (not part of pytest)
so a run can take minutes without slowing CI.

### Layout

```
backend/scripts/mcp_eval/
├── README.md
├── runner.py              # entry point: `python -m scripts.mcp_eval.runner`
├── tasks.yaml             # 8-task suite (frozen)
├── baseline_agent.py      # raw-paste: dumps top-N notes/issues into prompt
├── mcp_agent.py           # tool-using: only system prompt + jarvis tools
├── judge.py               # blind quality scorer (LLM-as-judge)
├── fixtures/
│   ├── workspace/         # tiny synthetic workspace (notes + jira)
│   └── ground_truth/      # expected facts per task
└── reports/               # output: Markdown + JSON per run
```

### Task suite (`tasks.yaml`)

8 tasks chosen to cover every namespace and every cost class:

| # | Task prompt | Tools expected | Cost class |
|---|---|---|---|
| 1 | "Summarize what we know about ONB-142, including blockers." | `jarvis_jira_describe_issue`, `jarvis_jira_blockers_of` | medium |
| 2 | "Find my notes about session timeout problems." | `jarvis_search_memory` (or `_notes`) | medium |
| 3 | "Who is involved in the Onboarding project? List people and their connections." | `jarvis_graph_query`, `jarvis_graph_entity_detail` | cheap×N |
| 4 | "What are the top 3 risk items in the current sprint?" | `jarvis_jira_sprint_risk` | medium |
| 5 | "Did I make any decisions about authentication in the last two weeks?" | `jarvis_session_recent_decisions` | medium |
| 6 | "What's my preferred test framework and why?" | `jarvis_get_preferences` | cheap |
| 7 | "Group the open ONB issues by topic." | `jarvis_jira_cluster_by_topic` | medium |
| 8 | "How is ONB-142 related to PROJECT-205?" | `jarvis_graph_path_between`, `jarvis_jira_describe_issue` | medium |

Each task in `tasks.yaml`:

```yaml
- id: 1
  prompt: "Summarize what we know about ONB-142, including blockers."
  ground_truth_facts:
    - "ONB-142 is about session expiring during onboarding wizard"
    - "blocked by ONB-130 (OAuth refresh token bug)"
    - "assigned to adam.nowak"
  baseline_context_files:
    - memory/jira/ONB/ONB-142.md
    - memory/jira/ONB/ONB-130.md
  max_tool_calls: 4
```

### Baseline agent

`baseline_agent.py`:

1. For each task, read every file in `baseline_context_files` (this is
   the "naive RAG" or "user pastes the whole folder" approach).
2. Build a single prompt: system + concatenated file bodies + question.
3. Call the configured LLM (default: local `ollama/qwen2.5:14b` for
   determinism; can override with `--model anthropic/claude-3-5-sonnet`).
4. Record: prompt token count, completion token count, wall-clock time,
   answer text.

### MCP agent

`mcp_agent.py`:

1. Spawns `python -m services.mcp --transport stdio --workspace fixtures/workspace`
   as a subprocess.
2. Implements a minimal MCP client (no SDK dependency: ~150 LOC over
   the JSON-RPC framing) so the harness can run without external apps.
3. For each task:
   - System prompt = "You have access to Jarvis tools. Prefer
     specialized tools (`jarvis_jira_*`, `jarvis_graph_*`) over generic
     `jarvis_search_memory`. Stop calling tools when you can answer."
   - Loop: model decides → call tool → feed result back → repeat until
     `stop_reason="end_turn"` or `max_tool_calls` reached.
4. Record: prompt token count *(initial system prompt only)*, all tool
   call payloads, total tool output tokens, completion tokens, tool
   names called, wall-clock time, answer text.

### Judge

`judge.py`:

- Blind scoring: receives `task.ground_truth_facts` + an answer, **does
  not know** which agent produced it.
- Asks the judge model: *"For each fact, did the answer correctly
  cover it? Respond with JSON: `{covered: int, total: int, hallucinations: int}`."*
- Default judge: `anthropic/claude-3-5-sonnet`. Override with
  `--judge ollama/llama3.1:70b` for fully-local eval (lower
  fidelity, documented).
- For each task pair, judge runs **once per agent** with answers in
  randomized order to reduce position bias.

### Report

After running all tasks for both agents, `runner.py` writes
`reports/<timestamp>.md`:

```markdown
# MCP eval — 2025-01-20T14:32:11Z

| Metric | Baseline | MCP | Δ |
|---|---:|---:|---:|
| Total input tokens | 142,300 | 18,400 | **−87%** |
| Total output tokens | 4,200 | 4,800 | +14% |
| Total cost (Claude Sonnet pricing) | $0.49 | $0.07 | **−86%** |
| Wall-clock total | 38s | 51s | +13s |
| Quality score (covered/total) | 38/48 (79%) | 41/48 (85%) | **+6pp** |
| Hallucinations | 4 | 1 | **−3** |

## Per-task breakdown

| # | Task | Baseline tokens | MCP tokens | Quality B → MCP |
|---|---|---:|---:|---|
| 1 | ONB-142 summary | 22,400 | 2,800 | 5/6 → 6/6 |
| 2 | Session timeout notes | 18,200 | 1,900 | 4/5 → 5/5 |
| ... |
```

Also emits machine-readable `reports/<timestamp>.json` for trend tracking.

### Acceptance gate

The harness is the **acceptance gate** for Step 24:

- ✅ `total_input_tokens(MCP) ≤ total_input_tokens(baseline) / 5`
  (≥5× input-token reduction).
- ✅ `quality_score(MCP) >= quality_score(baseline)` (no regression).
- ✅ `hallucinations(MCP) <= hallucinations(baseline)`.
- ✅ All 8 tasks complete within `max_tool_calls`.
- ✅ Re-running the harness twice on the same fixture produces results
  within ±10% on tokens and ±1 fact on quality (determinism check).

Failures block merge. The report is committed under
`docs/features/mcp-server/eval-runs/` so the trend is visible in git.

## Out of scope

- Public eval against real third-party datasets (LongBench etc.) — too
  noisy for an MVP signal.
- Continuous benchmark in CI — first run is opt-in (`make mcp-eval`).
  CI integration is a follow-up if eval runtime stabilizes < 2 minutes.
- Auto-tuning cost budgets from `mcp.jsonl` data — tracked as a future
  enhancement once we have a few weeks of real usage data.
